# Batch-Pool Skill Format Specification + Skill Generator · Design Document

> **Project Positioning**: This project (Skill Generator) is a standalone CLI tool for creating and maintaining Skill directories that conform to the Batch-Pool Engine specification.
>
> **Dependency Relationship**: Generator output → Engine consumption. The two are decoupled through a filesystem contract and evolve independently. The Generator **does not import** any code from the Engine, and the Engine **does not depend on** the Generator. Each maintains its own validation logic independently (copy, not share).
>
> **Contract Version**: v1.1 (defined in Part One of this document).
>
> **Compatibility Statement**: Skills generated by Skill Generator v1.x are compatible with Batch-Pool Engine v1.x. Minor versions are backward-compatible; Major versions may introduce breaking changes. Run `batch-pool validate --strict` to check compatibility with the latest engine contract.
>
> **v1.1 change**: Loop perspectives are now indexed in `config.yaml` (name / label / file / needs_context) with their critique prompt bodies stored in `perspectives/*.md` — symmetric with how tasks index into `references/*.md`. The Engine must resolve `perspective.file` the same way it resolves `task.file`. Inline `prompt:` perspectives remain accepted by readers for backward compatibility.

## Part One: Skill Format Specification v1.1

### 1. Directory Structure

```
my-skill/
├── SKILL.md              # Single entry point — metadata + task index
├── config.yaml           # Runtime configuration — concurrency params + synthesis strategy + Loop index
├── references/           # Task details — one file per task
│   ├── task-01.md
│   ├── task-02.md
│   └── task-03.md
└── perspectives/         # Loop critique prompts — one file per perspective
    ├── correctness.md
    └── evidence-quality.md
```

**Hard Constraints**:
- Batch-Pool only recognizes `SKILL.md` at the directory root
- `references/` is not a mandatory name — whatever `SKILL.md`'s `tasks[].file` points to is what matters
- `perspectives/` is likewise conventional — whatever `config.yaml`'s `loop.perspectives[].file` points to is what matters
- Using `references/` and `perspectives/` is recommended as a convention
- **On engine startup**, a core subset of `validator.validate()` is implicitly called (checking only: SKILL.md exists, YAML is parseable, tasks is non-empty, all task.file paths exist). If validation fails, the engine exits immediately and outputs `Skill format invalid, run batch_pool validate <path> for details`

---

### 2. SKILL.md Complete Schema

```yaml
---
# ── Required ──
name: string                 # Skill name, kebab-case, e.g. "code-security-audit"
description: string          # One-sentence description. LLM can understand the skill's purpose from this
version: "1.0"               # Semantic versioning

# ── Required: Task List ──
tasks:
  - id: string               # Unique identifier, kebab-case. e.g. "dependency-scan"
    file: string             # File path, relative to skill directory. e.g. "references/dependency-scan.md"
    label: string            # Human-readable label. e.g. "Dependency Chain Vulnerability Scan"
    priority: high|medium|low  # Scheduling priority

  - id: string
    file: string
    label: string
    priority: high|medium|low

  # ... at least 1 task, no upper limit

# ── Optional ──
category: string             # Category the skill belongs to, used for organizing multi-skill retrieval
tags: [string]               # Free-form tags
author: string               # Author
output_schema: string         # JSON structure description of the final artifact (multi-line text)
---
```

**Design Principles**:

- **SKILL.md only describes "what to do", not "how to do it".** Runtime configuration such as `workers`, `max_retries`, and loop perspectives all live in `config.yaml`.
- **The `file` field of a task is a relative path.** Resolved as `skill_dir / task.file`.
- **`priority` does exactly two things:** (1) pre-sorting when the Executor submits tasks (high priority enters the queue first); (2) sorting output by priority when the Synthesizer merges results.
- **`output_schema` defines how all task outputs combine into the final artifact.** The Engine uses it to decide the synthesis strategy; independent users use it to understand the artifact structure. The format is a multi-line string — it can be a JSON Schema fragment, a structured description, or an example. Optional but strongly recommended.
- **There is no `handler` field.** Batch-Pool v1 has only one handler: the LLM pipeline. Tasks do not need to declare one themselves.

### 3. config.yaml Complete Schema

```yaml
# ── Execution Configuration ──
execution:
  default_model: string       # Default model, e.g. "deepseek-v4-pro[1m]"
  max_tokens: int             # Max tokens per LLM call, default 4096
  workers: int                # Number of concurrent workers, default 4
  per_task_timeout: float     # Per-task timeout in seconds, default 120.0
  max_retries: int            # Retry count on failure, default 2

# ── API Pool Configuration ──
api_pool:
  max_concurrent_per_key: int # Concurrent slots per key, default 5

# ── Synthesis Configuration ──
synthesis:
  strategy: merge             # v1 only supports merge
  sort_by: priority           # priority | category | none
  output: synthesized.json    # Synthesis result filename

# ── Loop Configuration ──
loop:
  enabled: true               # Whether Loop is enabled
  max_iterations: int         # Max iteration rounds, default 3
  max_parallel_items: int     # Max items processed in parallel per round, default 8
  stop_condition: no_changes  # no_changes | max_iterations

  perspectives:               # Index of critique perspectives (v1.1)
    - name: string            # kebab-case ASCII id, e.g. "correctness" — drives the filename
      label: string           # Human-readable display name (may be non-ASCII), optional
      file: string            # Path to the critique prompt, e.g. "perspectives/correctness.md"
      needs_context: bool     # Whether global context (viewing other items) is needed, default false

  refine_prompt: string       # Revision instruction. All critiques are injected → LLM outputs revised version
```

**Design Principles**:

- **config.yaml is optional.** If missing, all defaults are used; the framework does not error.
- **Every field has a default value.** Users need only a SKILL.md at minimum to run.
- **Perspectives are an index (v1.1).** The critique prompt body lives in `perspectives/*.md` — symmetric with how tasks index into `references/*.md`. `config.yaml` holds only the runtime index (name / label / file / needs_context), keeping it free of large multi-line prompt blocks. The Generator always emits this file-based form.
- **Backward compatibility.** Readers (validator + engine) still accept a legacy inline `prompt: string` in place of `file:`. If both are present, `file:` wins. A bare built-in `name` with neither is resolved by the engine's built-in library.

### 3a. perspectives/*.md Specification (v1.1)

**A perspective file is the complete critique instruction, sent in its entirety to the LLM** — the same contract as `references/*.md`. It is not analyzed section-by-section; the whole file is the prompt.

```markdown
# [Perspective Label]

[The critique instruction: what single question this perspective asks about
the combined task output, and what JSON it should return.]
```

- The `# [Label]` heading is human-readable; including it in the prompt is harmless.
- The filename stem is derived from the perspective `name` (kebab-case ASCII). Pure-non-ASCII names fall back to a positional filename (`perspective-01.md`); the human-readable name is preserved in `config.yaml`'s `label`.
- Multiple agents can each own and edit one perspective file independently.

---

### 4. references/*.md Specification

**This is the actual content of a task, sent in its entirety to the LLM as a prompt.** Therefore it must be a self-contained instruction that lets the LLM complete the task independently.

```markdown
# [Task Title]

## Background
[One to two sentences explaining the background and purpose of this classification task]

## Analysis Dimensions
1. [Description of dimension 1]
2. [Description of dimension 2]
3. ...

## Output Format
[Explicitly require the JSON schema the LLM should output]
{
  "category": "...",
  "findings": [
    {
      "id": "...",
      "title": "...",
      "severity": "LOW|MEDIUM|HIGH|CRITICAL",
      "description": "...",
      "location": "...",
      "remediation": "..."
    }
  ],
  "score": 0-100,
  "summary": "..."
}

## Evaluation Criteria
- 90-100: [what condition]
- 70-89: [what condition]
- 50-69: [what condition]
- <50: [what condition]
```

**Field Contract (Hard Constraints)**:
- **Additive**: Custom fields may be added at the JSON root level (e.g., `cvss_score`, `cwe_id`). The Synthesizer will pass them through — it will not discard them just because it doesn't recognize them.
- **Non-subtractive**: Deleting the two core fields `findings` and `summary` is strictly prohibited. The `score` field may be set to `null` if a task does not use it, but keeping it is recommended to maintain consistent report structure.
- **Missing Field Handling**: If a task's output is missing the `findings` field, the Synthesizer logs a WARNING and sets that task's result to an empty list, without blocking the overall synthesis.
- **Cross-task Field Inconsistency Handling (Union Schema)**: If task A returns `cvss_score` but task B does not, the Synthesizer preserves the field on all items, filling missing values with `null`. Loop perspectives (e.g., severity-calibration) depend on these fields for cross-judgment — discarding non-uniform fields would weaken the basis for critique.

**Writing Principles**:

1. **Self-contained.** The LLM can only see the content of this one file (+ JSON output instructions). Do not reference another reference file inside a reference — the LLM cannot see it.
2. **Output format is a contract.** The Synthesizer merges results by task_id. If your output format differs from other tasks, fields will be lost during synthesis.
3. **Evaluation criteria must be concrete.** "Good architecture" is meaningless. "Authentication is independent of business logic; data flows have clear security boundaries" is actionable.
4. **Do not have semantic overlap with other tasks.** If two tasks both ask the LLM to find SQL injection, they will produce duplicate findings. Differentiate dimensions — one checks injection, one checks permissions, one checks dependencies — each looks at its own thing.
5. **Finding IDs are maintained by the user.** The skeleton produced by the generator starts `id` with the placeholder `$TASK_ID_UPPER-001`. When filling in the file, the user should increment IDs manually (e.g., `ARCH-001`, `ARCH-002`...). The generator does not auto-number, and the runtime does not validate continuity — IDs are solely for human reading and report referencing.

### 5. Task Splitting Guide

This is the most error-prone part of Skill design.

**Good Splitting (Clean Orthogonal Dimensions)**:

```
references/
├── architecture-review.md   ← Auth/authz/data flow boundaries/network encryption
├── dependency-scan.md       ← Third-party library versions/CVE/license compliance
├── injection-analysis.md    ← SQL/XSS/Command/SSTI injection
├── permission-audit.md      ← RBAC/least privilege/privilege escalation risks
└── data-leak-detection.md   ← Sensitive info leakage in logs/error messages/API responses
```

Each task looks at different things. They run concurrently without conflict.

**Bad Splitting (Semantic Overlap)**:

```
references/
├── security-overview.md     ← "Comprehensive security audit"
├── vulnerability-scan.md    ← "Scan all known vulnerabilities"
└── owasp-top10.md           ← "Check against OWASP Top 10"
```

All three tasks will look for injection. All three will look for XSS. After running, you will find three findings all pointing to the same line of code.

**Judgment Criterion: If the `id` prefixes of `findings` in two tasks' "output formats" are different (e.g., INJ-001 vs PERM-001), they are most likely orthogonal.**

**Don't be afraid to split finely.** Five focused 200-line tasks produce far higher quality output than one vague 1000-line task. The concurrent engine is designed for fine-grained tasks.

## Part Two: Skill Generator

### 1. Positioning

**Skill Generator is a CLI tool that helps users create a spec-compliant skill directory from scratch.**

What it does NOT do: define domain knowledge for you. What it DOES do: format, complete, and validate, so you can focus on writing task content.

#### Mode B: From YAML Definition File

```bash
$ python -m batch_pool new --from skill-definition.yaml
```

```yaml
# skill-definition.yaml — minimal input file
name: code-security-audit
description: Perform multi-dimensional security audit on codebase
category: security

tasks:
  - id: dependency-scan
    label: Dependency Chain Vulnerability Scan
    priority: high
  - id: architecture-review
    label: Architecture Security Audit
    priority: high
  - id: injection-analysis
    label: Injection Vulnerability Analysis
    priority: high
  - id: permission-audit
    label: Permission Model Audit
    priority: medium
  - id: data-leak-detection
    label: Data Leak Risk Detection
    priority: medium

# Optional: specify Loop perspectives
perspectives:
  - correctness
  - completeness
  - actionability
  - consistency
  - severity-calibration
```

**Perspective Resolution Rules**:

- If an element in the list is a **string** (e.g., `correctness`), the generator looks up the corresponding `prompt` and `needs_context` from the built-in perspective template library (Part Two, Section 5) and auto-completes it into a full `config.yaml` entry.
- If an element is a **dict** (e.g., `{name: custom-review, prompt: "...", needs_context: false}`), the generator uses the user-provided complete definition directly, without consulting the template library.
- String names **not matched in the template library** cause the generator to error and display the list of valid names — preventing silent perspective drops.
- **Deduplication and Override Rules**: The generator processes the perspective list in order, and **later perspectives with the same name override earlier ones** (whether string shorthand or dict definition). If a user intentionally writes two perspectives with the same name, the generator does not error, only logs an INFO message — the user is responsible for their own configuration. Deduplication is based solely on the `name` field; `prompt` content is not checked for consistency.
- **Task order is preserved**: The generator writes tasks into `SKILL.md` in the order they appear in the definition file's `tasks` list, without reordering by `priority`. `priority` is used only for runtime scheduling and synthesis sorting, and does not affect display order in SKILL.md.

### 4. Reference Skeleton Template

The generator will not write analysis dimensions for you — it does not know your domain. But it will generate a skeleton with clear placeholders to guide your filling.

**Template Rendering Approach**: Use Python's built-in `string.Template`, **without introducing a Jinja2 dependency**. Placeholders use `$VARIABLE` in UPPER_SNAKE_CASE style, distinguishing them from curly braces in the Markdown body. The template variable dict injected by the generator contains:

- `$TASK_ID` — the task's raw id (e.g., `dependency-scan`)
- `$TASK_ID_UPPER` — task id in UPPER CASE (e.g., `DEPENDENCY-SCAN`)
- `$TASK_LABEL` — task label (e.g., `Dependency Chain Vulnerability Scan`)

```markdown
# $TASK_LABEL

## Background
<!-- TODO: Describe the background and purpose of this classification task. 1-2 sentences. -->

## Analysis Dimensions
<!-- TODO: List specific check items. Each dimension should be an actionable judgment criterion. -->
1. <!-- TODO -->
2. <!-- TODO -->
3. <!-- TODO -->

## Output Format
<!-- Keep the following JSON schema unchanged, or adjust fields based on task characteristics -->
{
  "category": "$TASK_ID",
  "findings": [
    {
      "id": "$TASK_ID_UPPER-001",
      "title": "...",
      "severity": "LOW|MEDIUM|HIGH|CRITICAL",
      "description": "...",
      "location": "file:line_number",
      "remediation": "..."
    }
  ],
  "score": 0-100,
  "summary": "..."
}

## Evaluation Criteria
<!-- TODO: Define scoring criteria -->
- 90-100: <!-- TODO -->
- 70-89: <!-- TODO -->
- 50-69: <!-- TODO -->
- <50: <!-- TODO -->
```

**Why not use an LLM to auto-generate reference content?** The LLM does not know your codebase, your risk appetite, or your compliance requirements. The skeleton provides format guidance — the content is human judgment.

### 5. Built-in Perspective Template Library

The generator includes a set of common critique perspectives that users can reference by name (e.g., `perspectives: [correctness, completeness]`):

| Perspective Name | Purpose | needs_context |
|--------|------|---------------|
| `correctness` | Factual accuracy — do the referenced code/data truly exist? Are judgments correct? | false |
| `completeness` | Completeness — are relevant risks/issues/edge conditions missed? | false |
| `actionability` | Actionability — are remediation suggestions concrete and executable? Can a non-expert understand them? | false |
| `consistency` | Report consistency — does this contradict other findings? Are severity ratings consistent? Are there duplicates? | true |
| `severity-calibration` | Severity calibration — are LOW/MEDIUM/HIGH/CRITICAL ratings reasonable compared to other findings? | true |
| `evidence-quality` | Evidence quality — is the supporting evidence for findings sufficient? Any that "feel like" issues but lack evidence? | false |
| `false-positive-check` | False positive check — could this finding be a false positive? | false |

Users can override or extend — write custom `prompt` entries in `config.yaml`'s `perspectives`.

```bash

python -m batch_pool new

python -m batch_pool new --from my-skill.yaml

**Interactive UX Detail**: When the user enters a task id, the generator validates in real time whether it is valid kebab-case. If not, it prompts the user to re-enter immediately — rather than waiting until all inputs are collected to report the error.

```python
# validator.py
import re

def validate_task_id(task_id: str) -> bool:
    """Lowercase letters, digits, and hyphens only; must not start or end with a hyphen."""
    return bool(re.fullmatch(r'^[a-z0-9]+(?:-[a-z0-9]+)*$', task_id))
```

**Perspective Deduplication Implementation**: Uses a dict internally; iterates in order, later entries override earlier ones:

```python
def _resolve_perspectives(raw_list: list) -> list:
    resolved: dict[str, dict] = {}
    for entry in raw_list:
        if isinstance(entry, str):
            p = _load_from_builtin(entry)  # Load from template library
            if p:
                resolved[p["name"]] = p
        elif isinstance(entry, dict) and "name" in entry:
            resolved[entry["name"]] = entry
    return list(resolved.values())
```

### 9. Relationship with the Engine

```
Skill Generator                    Batch-Pool Engine
─────────────                       ──────────────
Creates skill directory            Consumes skill directory
Defines tasks (writes SKILL.md)    Discovers tasks (reads SKILL.md)
Configures Loop perspectives       Executes Loop (reads config.yaml)
  (writes config.yaml)
Fills reference content            Sends reference as prompt to LLM
Validates format correctness       Assumes format is correct (does not
                                     validate — guaranteed by Generator)
```

The Generator guarantees format correctness, the Engine guarantees execution correctness. Each handles its own responsibility, decoupled through the skill directory as a **filesystem interface**.

       describe what was changed; if nothing needed modification, set to empty string ""
```

Each `perspectives/*.md` holds that perspective's critique instruction, e.g.
`perspectives/correctness.md`:

```markdown
# Correctness

Critique this analysis from the perspective of factual accuracy.
Do the referenced code lines exist? Does the described behavior
actually occur? Are there factual errors in the technical judgment?
Only point out genuinely problematic areas; do not fabricate issues.
Respond with JSON: {"has_issues": bool, "issues": [{"point": "...", "severity": "critical|major|minor"}]}
```

## Part Four: --llm Single LLM Generation Mode

### 1. Positioning

Currently, the Generator's `new` command produces **TODO skeletons** — reference files contain only placeholders, with content filled manually by humans.

**`--llm` mode**: The user describes requirements in natural language → 1 LLM call → produces a complete, engine-ready Skill directory with correct formatting. Fully automated, zero human intervention.

```
User natural language description
       │
       ▼
┌─────────────────────────────┐
│  Prompt Builder              │  ← System = distilled format spec + splitting guide
│  + Adapter                   │     User   = user requirements + JSON output schema
│  + Parser (fault-tolerant +  │
│    auto-correct)             │
│  + Validator (hard checks)   │
└─────────────┬───────────────┘
              │ 1 LLM call
              ▼
   validate_basic() passes → Skill directory (engine-ready)
   fails → error reported + generated files preserved for debugging
```

**Quality Assurance Model**: Not reliant on human review — relies on two layers of hard constraints:

| Layer | Mechanism | What It Guarantees |
|----|------|---------|
| Parser | Auto-correct kebab-case / fill defaults / extract JSON | Structurally complete, no missing fields |
| Validator | `validate_basic()` 6 fatal checks | SKILL.md parseable, task.file exists, no duplicate ids, valid priority |

Parser + Validator passes = Engine's `_check_contract()` will definitely pass. The Generator and Executor share the same format contract.

**What is NOT guaranteed (and does not need to be)**: Semantic orthogonality between tasks, actionability of analysis dimensions, specificity of evaluation criteria. These are prompt soft constraints — if the LLM follows the spec, output will be acceptable; even if occasionally suboptimal, it does not affect the Engine's normal execution.

### 3. Prompt Design

#### 3.1 System Prompt

The full Skill format specification (Part One) is injected, distilled into instructions the LLM can directly execute. The key is having the LLM see **the same format constraints as the Engine**:

```
You are a Skill Designer. Create a complete, production-ready Batch-Pool
Skill from the user's natural language description.

## What is a Skill?
A Skill is consumed by the Batch-Pool Engine. The Engine:
1. Reads SKILL.md → discovers tasks by id, file path, priority
2. Sends each reference/*.md as an independent LLM prompt (tasks CANNOT
   reference each other — the LLM won't see other tasks)
3. Runs all tasks concurrently, synthesizes results by task id

Therefore your output MUST match the Engine's parsing contract EXACTLY.

## Task Splitting Rules (CRITICAL)
1. ORTHOGONAL DIMENSIONS. Each task analyzes a DIFFERENT dimension.
   Bad: "security-overview" + "vulnerability-scan" + "owasp-top10" → overlap
   Good: "injection-analysis" + "permission-audit" + "dependency-scan" → clean
   Test: can you describe each task's scope in one sentence without overlap?
2. SELF-CONTAINED. Every reference file is an independent LLM prompt.
   Do NOT write "see task X for details" — the LLM cannot see other tasks.
3. SPECIFIC SCORING. "Good architecture" is useless.
   "Auth middleware is applied to all routes except /public" is actionable.
4. FINDING ID PREFIX. Each task's findings use a unique id prefix derived
   from the task id (e.g., task "injection-analysis" → finding "INJ-001").
   This prevents ID collisions when the Engine synthesizes results.

## Reference File Structure
Every reference/*.md MUST follow this EXACT format:

# [Task Label]

## Background
1-2 sentences on what this task checks and why.

## Analysis Dimensions
1. [Specific, actionable check]
2. [Specific, actionable check]
3. ... (at least 3 dimensions)

## Output Format
{
  "category": "<task-id>",
  "findings": [
    {
      "id": "<PREFIX>-001",
      "title": "one-line summary",
      "severity": "LOW|MEDIUM|HIGH|CRITICAL",
      "description": "detailed explanation",
      "location": "file:line or component name",
      "remediation": "concrete fix steps"
    }
  ],
  "score": 0-100,
  "summary": "one-paragraph overall assessment"
}
You may ADD custom fields (e.g. cvss_score, cwe_id).
You MUST NOT remove any standard field.

## Evaluation Criteria
- 90-100: [concrete condition]
- 70-89:  [concrete condition]
- 50-69:  [concrete condition]
- <50:    [concrete condition]

## Available Perspectives
correctness | completeness | actionability | consistency |
severity-calibration | evidence-quality | false-positive-check
```

#### 3.2 User Prompt

```
Create a Batch-Pool Skill for this requirement:

<user input>

Respond with ONLY a JSON object (no markdown, no explanation):
{
  "name": "kebab-case-name",
  "description": "one sentence",
  "version": "1.0",
  "category": "single-word",
  "tags": [],
  "author": "",
  "tasks": [
    {
      "id": "kebab-case-id",
      "label": "Human-Readable Label",
      "priority": "high|medium|low",
      "reference": "# Label\n\n## Background\n...\n\n## Analysis Dimensions\n1. ...\n2. ...\n3. ...\n\n## Output Format\n{...}\n\n## Evaluation Criteria\n..."
    }
  ],
  "perspectives": ["correctness", "completeness", ...]
}

RULES:
- name and all task ids: STRICT kebab-case (lowercase + hyphens, no spaces)
- At least 2 tasks, recommended 3-7
- Each reference is a COMPLETE, self-contained markdown document
- Analysis dimensions: at least 3, each a single concrete check
- Scoring criteria: all 4 tiers filled with specific conditions
- Perspectives: pick 3-5 from the available list
- Labels: use the same language as the user's input
```

#### 3.3 Why a Single Call

A single call allows the LLM to see the full picture when splitting tasks — what dimension each task covers, what id prefix to use, how priority is distributed — these decisions require global awareness. If calls were split, the LLM for each call would not see other tasks, and the resulting dimensions would inevitably overlap.

The trade-off is a longer single prompt (~3-8K tokens), but modern model context windows handle this with ease.

### 5. Output Parsing and Validation

#### 5.1 Flow

```
LLM Response (str)
  │
  ├─ 1. Extract JSON (supports ```json fences and bare JSON)
  │     Failure → error and exit, save raw response
  │
  ├─ 2. Parse JSON → field validation + auto-correction
  │     - task.id not kebab-case → auto-convert, log WARNING
  │     - Missing version → "1.0", missing category → "general"
  │     - Missing priority → "medium", missing tags/author → empty
  │
  ├─ 3. Content inspection (non-blocking)
  │     - Does reference contain all four required sections?
  │     - Does findings JSON schema retain standard fields?
  │     Missing → log WARNING, do not block
  │
  └─ 4. Write + validate_basic()
         Passes → print OK + file listing
         Fails → print ERROR details + preserve generated files for debugging
```

#### 5.2 Fault Tolerance Table

| Issue | Behavior | Blocking? |
|------|------|--------|
| Non-JSON response | Error, save raw response to `last_response.txt` | 🔴 ERROR |
| JSON missing `name` or `tasks` | Error, cannot generate | 🔴 ERROR |
| `tasks` is empty array | Error | 🔴 ERROR |
| `task.id` is not kebab-case | Auto-convert (`SQL Injection` → `sql-injection`) | 🟡 WARNING |
| Missing `version` / `category` / `priority` / `tags` / `author` | Fill defaults | ⚪ Silent |
| Reference missing a section | Log WARNING, fill TODO placeholder | 🟡 WARNING |
| Reference missing standard JSON fields | Log WARNING | 🟡 WARNING |
| Response truncated | Retry once with `max_tokens` + 50% | 🔴 ERROR |
| `validate_basic()` fails | Error, preserve files | 🔴 ERROR |

### 6. Contract Relationship with the Engine

```
Skill Generator (--llm)              Batch-Pool Engine
─────────────────────────            ─────────────────
Generates SKILL.md                   Parses SKILL.md
  name: "my-skill"        →           task.id = "injection-analysis"
  tasks:                              task.file = "references/injection-analysis.md"
    - id: injection-analysis          task.priority = "high"
      file: references/...
      priority: high
                                      Engine._check_contract() checks:
Generates references/*.md             - SKILL.md exists and YAML is parseable
  ## Output Format                     - tasks is non-empty
  { "findings": [...],               - each task.file points to existing file
    "score": 0-100,                   - no duplicate task ids
    "summary": "..." }                - priority ∈ {high, medium, low}
        │
        │  These 5 checks = the scope of validate_basic()
        │  Generator passes = Engine will definitely pass
        ▼
  Engine runs concurrently
```

**Core Principle**: The Generator and Engine each maintain the same 5 validation checks independently, but the Generator's `validate_basic()` is stricter than the Engine's `_check_contract()` (one extra check: id non-empty). If the Generator passes, the Engine will never encounter a format error.
